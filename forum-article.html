CHAPTER 01

智能体循环 THE AGENT LOOP

真正的 agent 起点，是把真实工具结果重新喂回模型，而不只是输出一段文本。

问题：模型能思考但不能动手，不能打开文件、不能跑命令、不能观察报错，每一步都需要人复制粘贴结果。

方案：给模型接上工具，让思考 → 调用 → 拿结果形成自动循环。这就是最小的 Agent Loop——整个系统的心脏。

1
用户消息
进入 messages
2
调用模型
发送 messages + tools
3
看 stop_reason
是 tool_use 就继续
4
执行工具
run_tool(block)
5
回写结果
tool_result → messages
💡 关键洞察
消息历史不是聊天记录展示层

在 Agent 里，messages 更像"下一轮工作输入"。assistant 的回复、工具的执行结果，都必须写回 messages。少了任何一步，模型就无法基于真实观察继续工作。

核心代码：整个循环只有五步——调模型、追加回复、判断 stop_reason、执行工具、回写结果。

1# s01: 最小 Agent Loop
2messages = [{"role": "user", "content": query}]
3
4def agent_loop(state):
5    while True:
6        # 1. 调用模型
7        response = client.messages.create(
8            model=MODEL, system=SYSTEM,
9            tools=TOOLS, messages=state["messages"])
10
11        # 2. 追加 assistant 回复
12        state["messages"].append({
13            "role": "assistant", "content": response.content})
14
15        # 3. 如果不是 tool_use，结束
16        if response.stop_reason != "tool_use":
17            return
18
19        # 4. 执行工具
20        results = []
21        for block in response.content:
22            if block.type == "tool_use":
23                output = run_tool(block)
24                results.append({"type": "tool_result",
25                    "tool_use_id": block.id, "content": output})
26
27        # 5. 工具结果写回消息历史 ← 这是 Agent 的关键一行
28        state["messages"].append({"role": "user", "content": results})

CHAPTER 02

工具使用 TOOL USE

主循环本身不用变复杂；工具能力靠一层清晰的路由面增长。

问题：循环跑起来了，但只有一个 bash 工具。所有操作都走 shell，路径逃逸和危险命令完全拦不住。

方案：为高频操作定义专用工具（read_file、write_file、edit_file），在工具层面做路径沙箱。新增工具只需新增一行路由，核心循环永远不变。

组件 s01（之前） s02（之后）
工具数量 1（仅 bash） 4（bash, read, write, edit）
分发方式 硬编码 TOOL_HANDLERS 字典（dispatch map）
路径安全 无 safe_path() 路径沙箱
Agent Loop 不变 不变
💡 关键洞察
加工具 = 加 handler + 加 schema，循环永远不变

dispatch map 本质就是一个字典：{tool_name: handler_function}。模型说"我要调 read_file"，代码就去表里找对应函数。

核心代码：工具注册表 + 路径沙箱。

1# s02 新增：工具注册表

6    "edit_file":  lambda **kw: run_edit(kw["path"], kw["old_text"], kw["new_text"]),
7}

11output = handler(**block.input) if handler else f"Unknown: {block.name}"

18    return path

CHAPTER 03

会话内规划 TODO WRITE

对多步骤任务来说，可见计划不是装饰，而是防止会话漂移的稳定器。

问题：工具多了能干的事也多了，但大任务做着做着方向就漂了——没有计划全凭即兴发挥。

方案：让 Agent 先写任务清单再动手，每完成一步更新状态。约束：同一时间最多一个任务 in_progress；连续 3 轮不更新计划就注入提醒。

⚙️ 设计原则
计划外显 = 防止会话漂移

不是任务系统，只是当前会话的计划外显。清单让模型每一步都知道自己在做什么、下一步该做什么。

CHAPTER 04

子智能体 SUBAGENT

把探索性工作移进干净上下文后，父 agent 才能持续盯住主目标。

问题：探索性工作（搜索代码库、读大量文件）的中间结果全留在上下文里，噪声越积越多，主任务反而做不好。

方案：把局部任务交给 Subagent 在独立上下文里做。Subagent 有自己的 messages 和 tools（没有 task 工具，防止递归），做完只返回摘要，不污染父 agent 上下文。

探索结果、搜索日志全部堆积
主线任务被噪声淹没

✓
Subagent 隔离 + 摘要回传
探索在干净子上下文完成
只把必要结论带回父 agent
父 agent 上下文保持精炼

核心代码：Subagent 独立 messages + 受限工具列表，最终只返回文本摘要。

CHAPTER 06

上下文压缩 CONTEXT COMPACT

压缩的目标不是删历史，而是保住连续性和下一步所需的工作记忆。

问题：读了大量文件、跑了很多命令，上下文不断膨胀。Token 都烧完了，活儿才干一半。

方案：三层压缩策略，保住任务连续性的同时给上下文腾出空间。

L1
大结果落盘
persist_large_output
全文写磁盘，只留预览
L2
旧结果占位
micro_compact
保留最近 3 个，更旧的替换
L3
整体摘要压缩
compact_history
消息历史过长时生成连续性摘要
⚠️ 注意
压缩 ≠ 删除历史

压缩后必须保住五类信息：当前目标、已完成动作、改过的文件、关键决定、下一步计划。丢了这些，等于打断了工作连续性。

核心代码：主循环新增微压缩 + 自动压缩 + 手动触发压缩。

CHAPTER 07

权限系统 PERMISSION SYSTEM

模型产生的执行意图，必须先通过清晰的权限门，再变成真正动作。

问题：模型可能幻觉出错误路径，写错文件、执行危险命令。意图直接变执行，非常危险。

方案：工具调用前必须先经过四级权限管道：deny rules → mode check → allow rules → ask user。

🚫
1. Deny Rules
sudo、rm -rf
→ 绝对禁止
🔒
2. Mode Check
plan / auto / default
→ 按模式过滤
✅
3. Allow Rules
安全操作规则匹配
→ 自动放行
🙋
4. Ask User
灰区操作
→ 交给用户确认
💡 关键设计
bash 命令是自由文本，四级管道之前先过一遍 BashSecurityValidator 正则检查

检查 shell 元字符、sudo、rm -rf、命令替换等危险模式。bash 比结构化工具更危险，所以有额外的 Step 0。

核心代码：主循环在执行工具前插入权限管道。

1# s07 agent_loop 变更

3for block in response.content:
4    decision = perms.check(block.name, block.input or {})
5
6    if decision["behavior"] == "deny":
7        output = f"Permission denied: {decision['reason']}"
8    elif decision["behavior"] == "ask":
9        if perms.ask_user(block.name, block.input):
10            output = handler(**block.input)
11        else: output = "Permission denied by user"
12    else:  # allow
13        output = handler(**block.input)
14

17    def check(self, tool_name, tool_input) -> dict:
18        # 1. deny rules  2. mode check  3. allow rules  4. ask user
19        return {"behavior": "allow|deny|ask", "reason": "..."}

CHAPTER 08

Hook 系统 HOOKS

Hook 让系统围绕主循环生长，而不是不断重写主循环本身。

问题：安全审计、自动 lint、操作日志，每加一个横切需求都要改主循环。越改越重，小改动就可能影响全局。

方案：主循环在关键节点暴露生命周期事件（PreToolUse / PostToolUse / SessionStart），附加行为写成独立脚本，通过 .hooks.json 配置注册。统一退出码：0 继续、1 阻止、2 追加信息。

⚙️ 设计原则
Hook 不替代主循环，只在固定时机做旁路扩展

3 个生命周期事件 + 统一退出码协议 = 无限扩展能力，主循环代码不受影响。

核心代码：主循环在工具执行前后插入 hook 调用。

CHAPTER 10

系统提示词 SYSTEM PROMPT

CHAPTER 11

错误恢复 ERROR RECOVERY

系统必须清楚自己此刻是在继续、重试，还是处于恢复流程。

问题：输出截断、上下文爆了、API 超时——遇到错误主循环直接停住，用户不敢再用。

方案：先对错误分类，再选恢复路径。三条路径各有独立重试预算（默认 3 次），全部耗尽才真正失败。

错误 1：输出截断
续写恢复
注入续写消息：
"不要重来，从中断点接着写"
最多重试 3 次
错误 2：上下文爆了
压缩恢复
auto_compact() 摘要压缩
缩短后重试
最多重试 3 次
错误 3：网络抖动
退避重试
指数退避 + 随机 jitter
backoff_delay()
最多重试 3 次

核心代码：三条恢复路径嵌入主循环。

11                # 恢复路径 2：prompt_too_long → 压缩重试
12                if "prompt" in str(e):
13                    messages[:] = auto_compact(messages)
14                    continue
15                # 恢复路径 3：网络抖动 → 退避重试
16                time.sleep(backoff_delay(attempt))
